## Title
Decision Details Page: Responsive Layout + Score Rounding

## TL;DR
Two product-facing fixes shipped on the compliance decision details page:
1. **Responsive layout** — the page now stacks vertically on smaller screens instead of squishing into a side-by-side two-column layout.
2. **Score rounding** — screening match scores now display as clean whole percentages (e.g., `87` instead of `87.2344123`).

## Context
The decision details page (`/detection/decisions/$decisionId`) is the primary review surface for compliance teams. The previous layout used a fixed `2fr / 1fr` two-column grid that broke on smaller viewports, making panels overflow or compress awkwardly. Separately, match scores rendered with excessive decimal precision, reducing readability.

## What shipped / changed

### 1. Responsive decision page layout
- **File:** `packages/app-builder/src/routes/_app/_builder/detection/decisions/$decisionId.tsx`
- The main content area now uses **named grid areas** via the new `tailwindcss-grid-areas` plugin.
- Behavior:
  - **Below XL breakpoint:** content stacks vertically — main content first, right-side panels (score, outcome, trigger object) below.
  - **XL and above:** side-by-side layout returns — main content left, aside panels right.
- The right-side panel group (Score + Outcome) also switched from `flex-col` with an `lg:flex-row` breakpoint to a simpler `flex-row`, letting the parent grid handle the vertical/horizontal behavior.

### 2. Wrapping search-input tags
- **File:** `packages/app-builder/src/components/Decisions/ScreeningDetail.tsx`
- The search-input field list (the row of chips showing name, DOB, timestamps, etc.) now uses `flex flex-wrap`.
- When there are many search values or the viewport is narrow, tags **wrap to multiple lines** instead of overflowing or truncating.

### 3. Score rounding on match lists
- **File:** `packages/app-builder/src/components/CaseManager/ScreeningCaseDetail/ScreeningCaseMatches.tsx`
- Score display changed from `score * 100` to `Math.floor(score * 100)`.
- Result: users see whole-number percentages (e.g., `92` not `92.3847412`).

### 4. Supporting changes
- Added `tailwindcss-grid-areas` (`^0.1.1`) to `package.json` and `bun.lock`.
- Registered the plugin in `packages/app-builder/tailwind.config.ts`.

## Risks & landmines
- **Grid-area plugin:** `tailwindcss-grid-areas` is a new build-time dependency. If the plugin version is incompatible with the project's Tailwind version, the build will fail. The lockfile (`bun.lock`) was updated to pin it.
- **Layout shift on XL breakpoint:** users on screens near the XL threshold (typically ~1280px) will see a noticeable reflow when resizing. This is intentional but worth watching in usability testing.
- **Score rounding is now floor, not round:** `Math.floor` means `89.9%` displays as `89`, not `90`. If business expects standard rounding (`Math.round`), this is a behavior difference to confirm.

## How to verify

1. **Responsive layout**
   - Navigate to any decision detail page (`/detection/decisions/<id>`).
   - Resize browser from wide → narrow:
     - Below XL: main content stacks above score/outcome/trigger panels.
     - Above XL: side-by-side layout appears.
   - Confirm no horizontal scroll or clipped panels at widths between ~1024px and ~1280px.

2. **Wrapping tags**
   - Find a decision with multiple search-input values (name, DOB, timestamp, etc.).
   - Shrink the viewport so the tag row would normally overflow.
   - Confirm tags wrap to additional lines rather than cutting off or causing scroll.

3. **Score rounding**
   - Open a case with screening matches (continuous screening review or case detail matches).
   - Confirm scores display as whole numbers (e.g., `76`, `93`) without decimal points.
