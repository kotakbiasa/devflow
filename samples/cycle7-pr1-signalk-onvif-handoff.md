## Title
PTZ PUT API Integration for External Control

## TL;DR
The plugin now supports controlling ONVIF PTZ cameras through standard Signal K HTTP PUT requests. External plugins, apps, and scripts can pan, tilt, zoom, stop, and return cameras to home position without using the plugin's proprietary WebSocket protocol. Version bumped to **1.3.0**.

## Context
Previously, PTZ camera control was only possible through the plugin's built-in webapp and internal WebSocket messaging. This created a barrier for other Signal K plugins or external automation tools that wanted to drive cameras programmatically. This change opens that integration point using Signal K's standard PUT mechanism, which is the same pattern used throughout the Signal K ecosystem for writing data.

## What shipped / changed
- **Three new PTZ control endpoints** registered under `vessels.self/sensors/camera/ptz.{move,stop,home}`
- **Runnable example script** at `examples/ptz-control.js` demonstrating the full API with built-in timeout and polling logic
- **README documentation** updated with HTTP `curl` examples and a code snippet for calling from another Signal K plugin
- **297 lines of new unit tests** covering handler registration, auto-connect behavior, validation, and error status codes
- **Restart hardening** — PUT handlers are now re-registered on every `start()` because Signal K deregisters them when the plugin stops
- **Version bump** from `1.2.0` to `1.3.0` in `package.json`

## How it works (map)
External clients send standard HTTP PUT requests to the Signal K server:

| Action | Path | What to send |
|--------|------|--------------|
| Move | `.../ptz/move` | Camera IP + speed vector (`x`, `y`, `z` from `-1` to `1`) + optional timeout in seconds (`1`–`300`) |
| Stop | `.../ptz/stop` | Camera IP (or just the IP as a bare string) |
| Home | `.../ptz/home` | Camera IP + optional speed (`0`–`1`) |

The plugin automatically connects to the camera on demand if it has been discovered but not yet initialized, using the credentials configured in the plugin settings. It returns standard Signal K PUT results with clear status codes:

- `200` — success
- `400` — invalid input (bad address, speed out of range)
- `404` — camera not discovered
- `405` — camera has no PTZ support
- `409` — no media profile selected
- `502` — camera connection or command failed
- `503` — plugin is restarting

## Risks & landmines
- **Restart behavior**: Signal K removes PUT handlers when a plugin stops. The code now re-registers them on every `start()`, but if you are testing plugin reloads, confirm handlers come back automatically.
- **Discovery prerequisite**: The camera must be discovered (via auto-discovery or manually configured) before PTZ commands will work. An undiscovered camera returns `404`.
- **Media profile required**: The camera must have an active media profile selected; otherwise commands return `409`.
- **Network timeouts**: Single requests time out after 5 seconds; asynchronous PUT polling times out after 10 seconds. Very slow networks may need adjustment in the example script.

## How to verify
1. **Run tests**: `npm test` — the new `test/ptz-put-handler.test.ts` suite should pass alongside existing tests.
2. **Check documentation**: Open `README.md` and confirm the new "PTZ PUT API" section appears with HTTP examples.
3. **Try the example**: Run `node examples/ptz-control.js` against a development server with `SIGNALK_URL` and `CAMERA_ADDRESS` set. You should see a demo sequence that pans, tilts, zooms, stops, and goes home.
4. **Test via curl**: Use the `curl` commands from the README against your server to verify the endpoints respond with `200` on success.
5. **Confirm restart resilience**: Restart the plugin and verify the handlers still work without a server restart.
