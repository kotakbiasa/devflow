## Title
Traceable Sprite Edits — Asset Lab Update

## TL;DR
Asset Lab now supports **targeted edits** on existing sprites. Users can select any generated asset, type a change request (for example, "add glowing toxic core"), and generate new variants that preserve the original character’s identity. The system tracks which asset was edited and what instruction was used, so edit history is auditable.

## Context
Previously, Asset Lab could generate new sprites from scratch or reprint existing ones in different poses. If a user wanted a small change (different armor, new accessory, color tweak), they had to regenerate from the original prompt and hope the AI remembered the design. There was no way to say “take this exact sprite and only change X.”

This PR adds **traceable sprite edits**: a first-class editing workflow that locks the reference image’s identity, pose, composition, and materials, then applies only the requested change.

## What shipped / changed

### Product behavior
- **Edit workflow**: On any asset card in the Assets page, users can click **Edit** to open an inline panel (`apps/app/components/asset-edit-panel.tsx`), type an instruction (max 500 characters), choose how many variants to generate (1–4), and submit.
- **Lineage tracking**: Every edited asset records its `sourceId` (the original asset) and the exact `editInstruction` used. Sliced sprite sheets also inherit this lineage.
- **Visual indicators**: Edited assets display an **“Edited variant”** badge and the edit instruction text directly on the card, so the history is visible without opening a detail view.
- **Prompt logic**: The AI prompt distinguishes between a normal pose reprint (“only change the pose and angle”) and a targeted edit (“change only what the targeted edit requests; preserve every unmentioned detail”). This prevents accidental drift in character design.

### Under the hood
- **Database migration** (`apps/api/prisma/migrations/20260723120000_asset_edit_lineage/migration.sql`): Adds `sourceId` and `editInstruction` columns to the `Asset` table, with a foreign key back to `Asset.id` so lineage is queryable.
- **API route** (`apps/api/app/v1/assets/[id]/slice/route.ts`): When slicing a sprite sheet, the new frame assets inherit `sourceId` and `editInstruction` from the parent sheet.
- **Generate route** (`apps/api/lib/asset-generate.ts`): Accepts `sourceId` and `editInstruction`, validates inputs before any provider spend, uploads the source image as a reference, and saves the lineage on every generated record.
- **Prompt builder** (`apps/api/lib/asset-prompt.ts`): New `editInstruction` branch in `spritePrompt` that injects the targeted-edit directive.
- **Tests** (`apps/api/lib/asset-generate-route.test.ts`, `apps/api/lib/asset-prompt.test.ts`): Cover validation errors, batch edits, and prompt construction.

## Risks & landmines
- **Provider drift**: If the AI ignores the “preserve every unmentioned detail” instruction, edited sprites may drift from the source design. The prompt is tuned, but final quality depends on the model (`google/nano-banana-2`).
- **Chained edits**: Because edited assets can be re-edited, long chains may accumulate drift. There is no depth limit enforced today.
- **Database migration**: The new columns are nullable with `ON DELETE SET NULL`, so deleting a source asset won’t break referential integrity, but the edit history will show a broken link.
- **Rate limits**: Edits count against the same hourly generation limit as new renders (`GENERATE_HOURLY_LIMIT`, default 30/hour).

## How to verify
1. **Generate a base asset**: Create a sprite in Asset Lab (any game, any pose).
2. **Open the editor**: Click the **Edit** button on the asset card. The inline panel should appear below the card.
3. **Submit a targeted edit**: Type something specific like “add shoulder armor” and choose 1 variant. Click **Apply**.
4. **Check lineage**:
   - The new asset card should show an **“Edited variant”** badge.
   - Below the subject, the edit instruction text should be visible.
   - In the database, the new row should have `sourceId` matching the original asset and `editInstruction` matching your text.
5. **Check sliced sheets**: If you edit a sprite sheet and then slice it, the sliced frames should inherit the same `sourceId` and `editInstruction`.
6. **Check validation**:
   - Submit an edit with an empty instruction → should fail with “editInstruction must not be empty”.
   - Submit an edit without selecting a source → should fail with “sourceId is required when editInstruction is set”.
   - Submit a 501-character instruction → should fail with the length error.
