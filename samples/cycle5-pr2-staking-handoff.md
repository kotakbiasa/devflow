## Title
PR #69 Handoff: Stake Deposit Flow Restored

## TL;DR
The **Stake Deposit** flow is back. Users can now stake existing Deposit tickets (Token IDs) directly from the Stake page, alongside the existing Stake RING option. The original Deposit creation flow remains sunset (no new deposits can be opened).

## Context
A previous design decision sunset new RING deposit creation. However, staking *existing* Deposit tickets was mistakenly removed along with it. This PR restores only the staking portion, keeping the deposit creation sunset intact.

## What shipped / changed

**Restored (product-facing)**
- A new **"Stake Deposit"** tab inside the Stake modal (`src/view/stake/_components/new/index.tsx`).
- A deposit selection list with checkboxes, showing Token ID, balance, and term progress (`src/components/deposit-list.tsx`, `src/components/deposit-item.tsx`).
- A two-step transaction flow: **Approve** (one-time) → **Stake** (`src/view/stake/_components/new/stake-deposit.tsx`).
- New contract hooks for deposit approval and staking (`src/view/stake/_hooks/stake.ts`).

**Still sunset (unchanged)**
- New RING deposit creation is still disabled. The Deposit page shows a sunset notice with a governance link.
- No new deposit reward preview or term selector was re-introduced.

## How it works (map)

```
Stake page → "New Stake" modal
    ├── Tab: Stake RING (existing flow)
    └── Tab: Stake Deposit (restored flow)
            ├── DepositList (checkboxes)
            │     └── DepositItem (Token ID, amount, progress bar)
            ├── Button: Approve (if first time)
            └── Button: Stake
                  ├── Approval tx (if needed)
                  └── Stake tx (stakeDeposits contract call)
```

## Risks & landmines

- **Tab animation glitch**: The transition between tabs inside the modal has a known blur/opacity animation issue (NextUI + Framer Motion). The code includes a `// todo` comment referencing upstream issues. The tabs switch correctly; only the animation is affected.
- **Two-transaction flow**: If the user has not approved the staking contract to move deposits, they must first sign an **Approve** transaction, then a **Stake** transaction. The button label and loading state change accordingly.
- **Indexing gate**: Staking is blocked until the contract indexing check passes (`useCheckWaitingIndexing`). This prevents premature transactions on newly deployed contracts.
- **Console logs**: The `useDepositStake` hook logs transaction parameters to the browser console. These should be removed or replaced with a proper logger before production if they are not desired.

## How to verify

1. Open the app and navigate to **Stake**.
2. Click **New Stake**.
3. Verify two tabs appear: **Stake RING** and **Stake Deposit**.
4. Select **Stake Deposit**.
5. Verify a list of active deposit tickets appears, each showing:
   - Token ID
   - Balance
   - A progress bar reflecting the deposit term
6. Select one or more deposits via checkbox.
7. If not previously approved:
   - Click **Approve**, confirm wallet signature.
   - Wait for approval confirmation, then button switches to **Stake**.
8. Click **Stake**, confirm wallet signature.
9. Verify **TransactionStatus** toasts show Approval → Staking → Success/Fail.
10. On success, the selection clears and the deposit list refreshes.

**Negative test**: Navigate to the old Deposit creation page. Verify the sunset notice is still present and no "Deposit in Wallet" or reward preview creation flow is exposed.
