## Title
Handoff Packet: Edit Button on Item Detail Pages (Issue #782)

## TL;DR
Users with edit permissions now see an **Edit** button directly on game item, PC character item, and NPC character item detail pages. The button links to the correct edit form for that item and stays hidden for unauthorized users. This change adds no extra page load time.

## Context
Item detail pages previously let users view item info and upload photos (for some roles), but offered no direct path to edit the item from the show page. Users had to manually construct edit URLs or navigate through list views. Issue #782 requested an Edit button, aligned with how other show pages behave in the application.

## What shipped / changed
- **New UI element**: An **Edit** button appears at the top of item detail pages for authorized users.
- **Permission gating**: The button only shows if the current user has the `can_edit` permission for that item scope (game-level for game items, character-level for PC/NPC items).
- **Consistent routing**: Each item type builds its own edit link automatically:
  - Game item: `#/games/{gameSlug}/items/{itemId}/edit`
  - PC item: `#/games/{gameSlug}/pcs/{characterId}/items/{itemId}/edit`
  - NPC item: `#/games/{gameSlug}/npcs/{characterId}/items/{itemId}/edit`
- **No performance cost**: The edit permission check uses the same cached permission data the app already fetches to load items. It does not trigger a new network request.
- **Fail-closed behavior**: If the permission check fails or errors, `canEdit` defaults to `false` and the button remains hidden.

## How it works (map)
1. When an item detail page loads, its controller (`GameItemController` or `CharacterItemDetailController`) checks the user's permissions via the app's `AccessStore`.
2. The controller exposes `canEdit` to the page component.
3. The page component (`GameItem` or `CharacterItem`) builds the correct edit URL for that item type and passes it, along with `canEdit`, to the shared `ItemDetailHelper`.
4. `ItemDetailHelper` wraps the `EditButton` in a conditional render. If `canEdit` is true, the button appears in the page header alongside the Back button.
5. Clicking Edit navigates the user to the edit form for that specific item.

## Risks & landmines
- **Route assumptions**: The button links to edit routes that must already exist in the application. If an edit route is missing for a specific item type, the button leads to a 404.
- **Stale permission cache**: The permission check relies on `AccessStore` cache. If permissions change mid-session, the button may not update until a full page refresh. (This matches existing behavior for photo upload gating.)
- **Translation**: The button label uses the translation key `character_page.edit`. If this key is missing in a supported language, users may see the raw key or an empty button. (Note: the codebase uses this key across multiple pages, so it is likely already translated.)
- **Signature change**: `ItemDetailHelper.render` now requires `editHref` and `canEdit` parameters. Any custom or future caller must pass these or the button will not render correctly.

## How to verify
- **Authorized user**: Log in as a user with edit access to a game. Open a game item detail page, a PC item detail page, and an NPC item detail page. Confirm the Edit button appears in the page actions area for all three.
- **Unauthorized user**: Log in as a user without edit access. Confirm the Edit button is absent on all item detail pages.
- **Navigation**: Click the Edit button on each page type and confirm you land on the correct edit form URL pattern shown above.
- **Performance**: Confirm page load time feels unchanged. No additional spinner or delay should appear before the item data loads.
- **Error state**: Simulate a permission check failure (e.g., network issue for the permissions endpoint). Confirm the page still loads the item, but the Edit button stays hidden.
