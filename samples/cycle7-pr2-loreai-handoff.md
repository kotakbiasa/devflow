## Title
PR-1456: `lore import` Now Respects Your Provider Environment Settings

## TL;DR
Fixed a bug where the standalone `lore import` command ignored the `LORE_WORKER_MODEL` environment variable. If you configured your environment for a non-Anthropic provider (such as OpenAI) using environment variables, the import command would still default to Anthropic's model and fail with API errors. Now it respects your environment configuration and routes to the correct provider automatically.

## Context
Some users configure their AI provider entirely through environment variables rather than config files. When they set `LORE_WORKER_API_KEY`, `LORE_WORKER_UPSTREAM`, and `LORE_WORKER_MODEL` together, the standalone `lore import` command was ignoring the model setting and falling back to Anthropic's default model. This caused API 404 errors because it was hitting the wrong endpoint with the wrong model ID.

The normal background worker already honored these environment variables, but the standalone import path had its own separate logic that skipped them entirely.

## What shipped / changed
- **Unified model selection**: Both the live worker and the standalone import command now share the same logic for reading `LORE_WORKER_MODEL`, eliminating the inconsistency.
- **Correct priority order**: When choosing a model for import, the system now checks in this order:
  1. The `LORE_WORKER_MODEL` environment variable (highest priority)
  2. Your `.lore.json` configuration file (`workerModel`)
  3. The current session model
- **Format support**: `LORE_WORKER_MODEL` accepts either `provider/model` (e.g., `openai/gpt-5.4-mini`) or just a bare model name (which defaults to Anthropic for backward compatibility).
- **Regression tests added**: New tests verify that the env var is honored during import and that the parser handles OpenRouter-style nested paths correctly.

## How it works (map)
Think of model selection as a checklist during import:
1. **Environment variable first**: If `LORE_WORKER_MODEL` is set, use it immediately. This lets you override everything else without editing files.
2. **Config file second**: If no env var, check `.lore.json` for an explicit `workerModel`.
3. **Session fallback**: If neither is set, use whatever model your current session is using.

Previously, step 1 was missing from the import command. Now it behaves exactly like the live worker.

## Risks & landmines
- **Low risk**: This only changes which model/provider gets selected during import. It does not change authentication, data extraction, or file formats.
- **Value format**: Ensure `LORE_WORKER_MODEL` is formatted correctly. Previously, a malformed or unexpected value was silently ignored; now it will be parsed and used. Double-check any scripts or documentation that set this variable.
- **Test isolation**: The codebase now cleans up this environment variable between tests, but if you run tests in a shared environment where this variable is set globally, it could affect outcomes. Isolated CI environments remain best practice.
